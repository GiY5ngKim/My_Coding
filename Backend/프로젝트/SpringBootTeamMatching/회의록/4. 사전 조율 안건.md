> **중간·후반부에 구조 갈아엎는 사태 방지**  
> **프론트–백 충돌 최소화**  
> **백엔드 내부 구현 난이도 폭증 방지**

# Ⅰ. 백엔드–프론트엔드 전체 회의 안건

(= “API 계약 + 인증 계약”)

> ❗ 여기서 정한 건 **중간에 바꾸면 일정 터짐**

---

## 1️⃣ API URL & HTTP Method 규칙

### 왜 필요한가

- 프론트는 URL을 코드에 직접 사용
    
- URL 변경 = 프론트 대량 수정
    

### 반드시 정할 것

- **도메인 기준 URL**
    
    - `/api/auth`
        
    - `/api/mates`
        
    - `/api/recruits/projects`
        
    - `/api/recruits/contests`
        
    - `/api/recruits/studies`
        
    - `/api/boards`
        
- CRUD별 HTTP Method 고정
    
    - 조회: GET
        
    - 생성: POST
        
    - 수정: PATCH
        
    - 삭제: DELETE
        

---

## 2️⃣ Request / Response JSON 구조

### 왜 필요한가

- 프론트 화면은 JSON 구조에 종속
    
- 필드명 하나 바뀌면 화면 깨짐
    

### 반드시 정할 것

- 필드명 규칙 (camelCase)
    
- 필수/선택 필드
    
- 응답에 포함할 최소 정보
    
- 리스트 응답 구조
    

#### 예시

```json
{
  "items": [...],
  "totalCount": 42
}
```

---

## 3️⃣ 인증 방식 & 토큰 정책

### 왜 필요한가

- 프론트와 백이 동시에 맞춰야 작동
    
- 나중에 바꾸기 매우 어려움
    

### 반드시 정할 것

- Google OAuth 사용
    
- 서버에서 이메일 도메인(@catholic.ac.kr) 검증
    
- **Access Token만 사용 (JWT)**
    
- 토큰 전달 방식
    
    - Authorization Header vs HttpOnly Cookie
        

---

## 4️⃣ 에러 응답 규약

### 왜 필요한가

- 프론트는 에러 코드로 화면 분기
    
- 메시지 문자열만 있으면 처리 불가
    

### 반드시 정할 것

- 공통 에러 포맷
    

```json
{
  "errorCode": "AUTH_401",
  "message": "로그인이 필요합니다"
}
```

- 에러 코드 네이밍 규칙
    

---

## 5️⃣ Enum 목록 & 고정 여부

### 왜 필요한가

- Enum은 프론트/백/DB 전부에 영향
    
- 중간 변경 비용 큼
    

### 반드시 정할 것

- Part Enum 최종 목록
    
- Role 여부
    
- 영문 대문자 고정
    
- 화면 표시용 한글 변환 방식
    

---

## 6️⃣ 페이지 초기 로딩 데이터 전략

### 왜 필요한가

- 프론트는 “페이지 단위 데이터”를 원함
    
- 백엔드는 “리소스 단위 API”를 만듦
    
- 이 차이를 안 맞추면 API 추가 요구 폭증
    

### 반드시 정할 것

- 홈 화면용 집계 API 허용 여부
    
    - 예: `/api/home`
        
- 상세 페이지에서 필요한 데이터 범위
    

---

## 7️⃣ 페이징 / 정렬 기준

### 왜 필요한가

- 리스트 서비스에서 필수
    
- 나중에 붙이면 API 구조 변경 발생
    

### 반드시 정할 것

- page / size / sort 파라미터 이름
    
- 기본 정렬 기준
    
- 응답에 totalCount 포함 여부
    

---

## 8️⃣ 인증/권한 실패 시 UX 흐름

### 왜 필요한가

- 401 / 403 처리 다름
    
- 프론트가 어떻게 반응할지 정해야 함
    

### 반드시 정할 것

- 401 → 로그인 페이지 이동
    
- 403 → 권한 없음 안내
    
- 삭제/수정 실패 시 UX 처리 방식
    

---

# Ⅱ. 백엔드 내부 회의 안건

(= “구현 철학 + 유지보수 전략”)

> ❗ 프론트는 몰라도 되지만  
> ❗ 안 정하면 **백엔드만 죽음**

---

## 1️⃣ Entity 설계 & 관계 방향

### 왜 필요한가

- DB 구조 변경 비용 큼
    
- 양방향 참조는 사고 확률 높음
    

### 반드시 정할 것

- 핵심 Entity 목록
    
- 연관관계는 **단방향 우선**
    
- 공통 필드(BaseEntity) 여부
    

---

## 2️⃣ DTO ↔ Entity 변환 책임 위치

### 왜 필요한가

- Controller 비대화 방지
    
- 코드 일관성 유지
    

### 반드시 정할 것

- 변환 위치: Service
    
- 수동 매핑 vs Mapper 사용 여부
    

---

## 3️⃣ 인증/인가 처리 구조

### 왜 필요한가

- Controller마다 검사하면 코드 중복
    

### 반드시 정할 것

- 인증: 필터(Spring Security)
    
- 인가: 어노테이션 or Service 로직
    

---

## 4️⃣ 예외 처리 전략

### 왜 필요한가

- 컨트롤러마다 try-catch 지옥 방지
    
- 프론트 에러 규약과 직결
    

### 반드시 정할 것

- GlobalExceptionHandler 사용
    
- 커스텀 예외 구조
    

---

## 5️⃣ 트랜잭션 기준

### 왜 필요한가

- 중간 실패 시 데이터 꼬임 방지
    

### 반드시 정할 것

- Service 레벨 @Transactional 기본
    
- 조회용 readOnly 트랜잭션 여부
    

---

## 6️⃣ 삭제 정책 (Soft vs Hard)

### 왜 필요한가

- 복구/통계/참조 무결성 문제 방지
    

### 반드시 정할 것

- Soft Delete 사용 여부
    
- deletedAt 컬럼 사용 여부
    

---

## 7️⃣ Docker & 배포 구조 (최소)

### 왜 필요한가

- 배포 단계에서 가장 많이 터짐
    

### 반드시 정할 것

- Dockerfile 구조
    
- 로컬/운영 설정 분리
    
- DB 컨테이너 포함 여부
    

---

## 8️⃣ Git 협업 규칙 (최소)

### 왜 필요한가

- 충돌/히스토리 꼬임 방지
    

### 반드시 정할 것

- 브랜치 전략 (main/dev/feature)
    
- PR 단위 기준
    
- 커밋 메시지 규칙
