## ✅ 줄별 역할/기능 해설

```yaml
spring:
  jpa:
    show-sql: false
    # Hibernate가 실행하는 SQL을 콘솔에 출력하지 않음.
    # 운영 환경에서 SQL 로그는 성능 저하 및 민감 정보 노출 위험이 있으므로 비활성화가 일반적.

    properties:
      hibernate:
        format_sql: false
        # SQL 포맷팅 비활성화.
        # show-sql이 false이므로 사실상 의미는 거의 없음.
        # 운영에서는 불필요한 포맷 비용을 줄이는 게 일반적.

logging:
  level:
    org.flywaydb: INFO
    # Flyway 마이그레이션 로그 레벨을 INFO로 설정.
    # 운영 환경에서는 DEBUG 대신 INFO/WARN 정도가 적절.
    # 마이그레이션 성공/실패 여부만 확인하는 수준.

app:
  cookie:
    secure: true
    # 쿠키에 Secure 플래그를 붙이도록 설정.
    # HTTPS 연결에서만 쿠키 전송.
    # 운영 환경에서는 반드시 true여야 함.

    same-site: None
    # SameSite=None 설정.
    # 크로스사이트 요청에서도 쿠키 전송 허용.
    # 단, SameSite=None이면 반드시 Secure=true여야 브라우저에서 허용됨.
```

# 🔴 P0 — 반드시 점검해야 할 부분

---

## 1️⃣ SameSite=None + Secure=true 조합은 HTTPS 필수

```yaml
secure: true
same-site: None
```

⚠️ 브라우저 정책:

- SameSite=None이면 반드시 Secure=true여야 함
    
- Secure=true면 HTTPS에서만 쿠키 전송
    

👉 운영 서버가 반드시 HTTPS여야 한다.

HTTP 환경에서 이 설정이면:

- 쿠키가 전송되지 않음
    
- 로그인 유지 실패
    

---

## 2️⃣ cookie 설정이 실제 코드에 반영되는지 확인 필요

application-local.yml과 동일하게:

```yaml
app:
  cookie:
    secure: true
    same-site: None
```

이 값이 실제 쿠키 생성 코드에서 읽혀서 설정되지 않으면  
이 파일은 아무 의미가 없다.

예:

```java
response.addHeader("Set-Cookie", ...);
```

여기서 secure/sameSite 설정이 이 값을 참조해야 한다.

# 🟡 P1 — 실무 권장

---

## 3️⃣ SameSite 전략 재검토 필요

SameSite=None은:

- 프론트와 API 도메인이 다를 때 필요
    
- SPA가 다른 도메인에서 실행될 경우 필요
    

하지만:

- 같은 도메인에서 서비스하면 Lax로도 충분
    

현재 구조가:

- 프론트 도메인 ≠ 백엔드 도메인 → None 필요
    
- 동일 도메인 → Lax 권장

운영 환경에서 show-sql=false는 필수.

# 📌 핵심 정리

✔ local / prod 설정 분리는 잘 되어 있음  
✔ Secure=true 설정은 반드시 HTTPS 전제

✔ cookie 설정이 실제 코드에 적용되는지 반드시 확인 필요
