## ✅ 줄별 역할/기능 해설

```sql
-- V1_init.sql
-- Flyway에서 사용하는 버전 기반 마이그레이션 파일.
-- V1__*.sql 형식으로 시작하는 파일은 애플리케이션 시작 시 자동 실행된다.
-- 이 파일은 DB 초기 스키마 생성을 담당하는 1번 버전 마이그레이션이다.

CREATE TABLE test_table (
    id SERIAL PRIMARY KEY,
    -- id 컬럼 생성
    -- SERIAL: PostgreSQL에서 자동 증가 정수 타입
    -- 내부적으로 sequence를 생성하고 DEFAULT nextval(sequence) 설정
    -- PRIMARY KEY: NOT NULL + UNIQUE + 자동 인덱스 생성

    name VARCHAR(50) NOT NULL
    -- name 컬럼 생성
    -- VARCHAR(50): 최대 50자 문자열
    -- NOT NULL: NULL 값 허용하지 않음
);

-- 위 구문 실행 시 PostgreSQL 내부 동작:
-- 1) test_table 생성
-- 2) id용 sequence 생성 (예: test_table_id_seq)
-- 3) id 기본값을 nextval(sequence)로 설정
-- 4) PK 인덱스 자동 생성

INSERT INTO test_table(name) VALUES ('hello');
-- test_table에 데이터 1건 삽입
-- id는 SERIAL이므로 자동으로 1이 들어감 (초기 상태라면)
-- name 컬럼에 'hello' 저장
```

## 🔴 P0 — 실무 기준 필수 점검 사항

### 1️⃣ Flyway 네이밍 규칙 확인

Flyway 기본 규칙:

```
V1__init.sql   ← 언더바 2개
```

현재 파일명이:

```
V1_init.sql
```

⚠️ 언더바 1개면 Flyway에서 인식 안 될 수 있음  
반드시 `V1__init.sql` 형식인지 확인 필요.

---

### 2️⃣ 운영 환경에서 테스트 데이터 삽입 금지

```sql
INSERT INTO test_table(name) VALUES ('hello');
```

실무에서는:

- migration 파일에 테스트 데이터 넣지 않음
    
- seed 데이터는 별도 migration or dev profile 전용 스크립트로 분리
    

졸업 프로젝트라면 괜찮지만,  
운영 배포 시에는 제거 권장.

---

## 🟡 P1 — 실무 권장

### 3️⃣ SERIAL 대신 IDENTITY 사용 고려 (PostgreSQL 10+)

SERIAL은 레거시 방식.

최신 권장:

```sql
id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
```

장점:

- 표준 SQL 문법
    
- sequence 관리 명확
    
- 향후 PostgreSQL 권장 방식
    

졸업 프로젝트라면 SERIAL도 충분.

---

### 4️⃣ 테이블 네이밍 전략

`test_table`은 테스트용 네이밍.

실제 도메인 테이블은:

```
app_user
project_post
study_post
```

처럼 도메인 기반 네이밍이 바람직.
