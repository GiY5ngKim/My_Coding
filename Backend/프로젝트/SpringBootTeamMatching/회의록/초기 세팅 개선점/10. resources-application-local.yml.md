## ✅ 줄별 역할/기능 해설

```yaml
spring:
  jpa:
    show-sql: true
    # Hibernate가 실행하는 SQL을 콘솔에 출력하도록 설정.
    # 개발 환경에서 디버깅에 유용.
    # 운영 환경에서는 보통 false.

    properties:
      hibernate:
        format_sql: true
        # SQL을 보기 좋게 줄바꿈/정렬하여 출력.
        # show-sql=true일 때 가독성 향상.

logging:
  level:
    org.flywaydb: DEBUG
    # Flyway 마이그레이션 실행 로그를 DEBUG 레벨로 출력.
    # 어떤 SQL이 실행되는지, 마이그레이션 상태가 어떤지 확인 가능.
    # 개발 환경 디버깅용.

app:
  cookie:
    secure: false
    # 쿠키에 Secure 플래그를 붙일지 여부.
    # false → HTTP에서도 쿠키 전송 가능.
    # 로컬(HTTP) 개발 환경에서는 false가 일반적.

    same-site: Lax
    # 쿠키 SameSite 속성 설정.
    # Lax → 대부분의 CSRF 공격 방어에 도움.
    # Strict보다 완화된 정책.
```

# 🔴 P0 — 반드시 이해하고 넘어가야 할 부분

---

## 1️⃣ cookie 설정이 실제로 코드에서 사용되는지 확인 필요

현재 설정:

```yaml
app:
  cookie:
    secure: false
    same-site: Lax
```

⚠️ 이 설정은 단순히 yml에만 존재하고  
코드에서 @ConfigurationProperties 등으로 읽어 사용하지 않으면 아무 의미가 없다.

즉,

- 실제 쿠키 생성 시
    
- secure / same-site를 적용하는 로직이 있어야 유효
    

👉 Jwt 발급 코드나 Cookie 생성 코드에서 이 값을 사용하는지 확인 필요.

---

## 2️⃣ show-sql은 실제 운영에서 절대 금지

지금은 local 프로파일이므로 문제 없음.

운영 환경에서:

- SQL 로그는 성능 저하
    
- 민감 정보 노출 위험

# 🟡 P1 — 실무 권장

## 3️⃣ Flyway DEBUG 로그는 local 전용으로 적절

```yaml
org.flywaydb: DEBUG
```

✔ local에서 마이그레이션 디버깅에 매우 유용  
✔ 운영에서는 INFO 또는 WARN 권장

---

## 4️⃣ SameSite=Lax는 합리적 기본값

쿠키 기반 인증을 쓴다면:

- Strict → 보안 강하지만 외부 리다이렉트 시 문제 발생 가능
    
- Lax → 일반적인 로그인 서비스에 적합
    

지금은 JWT 헤더 기반이면 큰 영향 없음.

## 5️⃣ Hibernate SQL 로그 레벨 통제 가능

대신 이런 방식도 있음:

```yaml
logging:
  level:
    org.hibernate.SQL: DEBUG
    org.hibernate.type.descriptor.sql.BasicBinder: TRACE
```

바인딩 값까지 확인 가능.
