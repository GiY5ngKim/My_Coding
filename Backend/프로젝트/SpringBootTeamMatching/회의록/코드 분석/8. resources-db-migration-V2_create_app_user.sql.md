## ✅ 줄별 역할/기능 해설

```sql
-- V2_create_app_user.sql
-- Flyway 마이그레이션 v2: 사용자 테이블(app_user) 생성 및 유니크 제약조건 추가

CREATE TABLE app_user (
    user_id BIGSERIAL PRIMARY KEY,
    -- user_id: PK 컬럼
    -- BIGSERIAL: 자동 증가 8바이트 정수 (sequence 기반)
    -- PRIMARY KEY: NOT NULL + UNIQUE + PK 인덱스 자동 생성

    provider VARCHAR(20) NOT NULL,
    -- provider: OAuth 공급자 식별자 (예: 'google')
    -- VARCHAR(20): 최대 20자 문자열
    -- NOT NULL: 반드시 값 존재

    provider_sub VARCHAR(255) NOT NULL,
    -- provider_sub: OAuth 공급자가 발급한 사용자 고유 식별자 (구글 idToken의 sub)
    -- VARCHAR(255): 최대 255자
    -- NOT NULL

    email VARCHAR(255) NOT NULL,
    -- email: 사용자 이메일
    -- NOT NULL

    email_verified BOOLEAN NOT NULL,
    -- email_verified: 이메일 검증 여부
    -- BOOLEAN NOT NULL: true/false 중 하나가 항상 들어가야 함

    created_at TIMESTAMP DEFAULT now(),
    -- created_at: 생성 시각
    -- TIMESTAMP: 타임존 정보 없는 timestamp (timestamp without time zone)
    -- DEFAULT now(): INSERT 시 값이 없으면 현재 시각으로 채움
    -- (주의: NOT NULL 제약이 없어서 NULL 저장도 이론상 가능)

    updated_at TIMESTAMP DEFAULT now()
    -- updated_at: 수정 시각
    -- DEFAULT now(): INSERT 시 값이 없으면 현재 시각으로 채움
    -- (주의: UPDATE 시 자동 갱신은 안 됨. "기본값"은 INSERT 시에만 유효)
);

ALTER TABLE app_user
    ADD CONSTRAINT uk_app_user_email UNIQUE (email);
-- email 컬럼에 유니크 제약조건 추가
-- 같은 이메일은 2번 저장 불가

ALTER TABLE app_user
    ADD CONSTRAINT uk_app_user_provider_sub UNIQUE (provider, provider_sub);
-- (provider, provider_sub) 조합에 유니크 제약조건 추가
-- 같은 provider에서 같은 sub로 중복 가입 불가
```

## 🔴 P0 — 반드시 수정/점검 (정합성/운영 장애 가능)

### 1️⃣ `created_at`, `updated_at`에 NOT NULL이 없다 (JPA 매핑과 불일치 가능)

#### 현재 스키마

```sql
created_at TIMESTAMP DEFAULT now(),
updated_at TIMESTAMP DEFAULT now()
```

#### 왜 문제?

너희 `Member` 엔티티는:

```java
@Column(name="created_at", nullable=false, updatable=false)
@Column(name="updated_at", nullable=false)
```

즉 JPA 관점에서는 NOT NULL을 전제로 한다.

DB는 NOT NULL이 없으니

- 직접 SQL로 넣으면 NULL 들어갈 수 있음
    
- 운영 중 데이터 정합성 깨질 수 있음
    

#### ✅ 수정안(실무 추천)

```sql
created_at TIMESTAMP NOT NULL DEFAULT now(),
updated_at TIMESTAMP NOT NULL DEFAULT now()
```

---

### 2️⃣ `updated_at DEFAULT now()`는 “자동 갱신”이 아니다

#### 왜 문제?

많이 착각하는 부분인데:

- DEFAULT now()는 INSERT 시 기본값
    
- UPDATE 때는 자동으로 바뀌지 않음
    

즉, 수정해도 updated_at이 그대로일 수 있음.

#### ✅ 실무 추천 해결책(택1)

**A) 앱 레벨(JPA Auditing)로 업데이트**

- 현재 이미 @LastModifiedDate 쓰고 있으니
    
- 코드에서 업데이트 때 값 세팅되면 DB는 기본값만으로도 큰 문제는 없음
    

→ 그래도 NOT NULL은 추천

**B) DB 트리거로 자동 갱신**

- 실무에서 DB 주도 업데이트 관리할 때 사용
    

---

### 3️⃣ `provider`, `provider_sub`는 조회 키이므로 인덱스는 사실상 OK

유니크 제약 자체가 인덱스로 생성되는 경우가 대부분이라 조회 성능은 대체로 문제 없음.

---

## 🟡 P1 — 실무 권장 (졸업프로젝트에서는 선택)

### 4️⃣ TIMESTAMP vs TIMESTAMPTZ (타임존)

현재:

```sql
TIMESTAMP
```

실무에서는 서버/DB/지역이 섞이면  
`TIMESTAMPTZ`(timezone 포함) 권장.

졸업 프로젝트(단일 리전)에서는 TIMESTAMP도 충분.

---

### 5️⃣ email_verified에 DEFAULT 추가 고려

현재는 NOT NULL만 있음:

```sql
email_verified BOOLEAN NOT NULL
```

삽입 시 반드시 값을 넣어야 함.

실무에서는 default false를 두기도 함:

```sql
email_verified BOOLEAN NOT NULL DEFAULT false
```

단, OAuth idToken 검증 결과를 그대로 저장할 거면 지금처럼 명시적으로 넣는 것도 괜찮음.

---

## 🔵 P2 — 후순위(운영 고도화)

### 6️⃣ provider를 enum 수준으로 제한 (DB CHECK 제약)

실무에서는 provider 값이 제한적이면:

```sql
ALTER TABLE app_user
ADD CONSTRAINT chk_provider CHECK (provider IN ('google'));
```

졸업 프로젝트에서는 생략 가능.

# 📌 최종 요약

- 테이블 구조는 `Member` 엔티티와 잘 맞음
    
- 다만 실무 기준으로는 **created_at/updated_at에 NOT NULL 추가가 최우선(P0)**
    
- `updated_at DEFAULT now()`는 자동 갱신이 아니라서  
    **JPA Auditing(현재 사용 중)으로 관리하는 게 정상**
