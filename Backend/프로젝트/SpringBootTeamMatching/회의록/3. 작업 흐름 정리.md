## 백엔드 실전 아키텍처 & 협업 가이드

---

## 0. 이 문서의 목적

이 문서는 다음을 **한 번에 이해**하기 위한 실전 정리본이다.

- 프론트 ↔ 백엔드는 **어떤 프로그램**으로 **어떤 흐름**으로 연결되는가
    
- JSON → DTO → Service → Entity → DB → Response DTO → JSON  
    이 흐름에서 **각 단계의 정확한 역할**
    
- OAuth + JWT 인증은 **왜 서버가 주도해야 하는지**
    
- DTO / Entity / Enum / 에러 포맷을 **왜 반드시 분리·고정해야 하는지**
    
- 협업에서 “이걸 안 정하면 100% 터진다”는 포인트들

## 2. 프론트–백 연동의 정확한 역할 분리

### 2-1. 브라우저(프론트)의 역할

- 화면(UI) 렌더링
    
- 사용자 입력 수집
    
- JSON 형태로 데이터 구성
    
- HTTP 요청 전송
    
- 응답(JSON, status code)에 따라 화면 분기
    

⚠️ **프론트는 절대 “보안 주체”가 아니다**

- JS는 개발자도구로 조작 가능
    
- API 요청은 curl/Postman으로 누구나 직접 보낼 수 있음
    

---

### 2-2. 서버(백엔드)의 역할

- 모든 요청의 **최종 승인자**
    
- 인증/인가 판단
    
- 비즈니스 규칙 강제
    
- DB 무결성 유지
    
- “신뢰 가능한 결과”만 응답
    

👉 **프론트에서 막아도 서버에서 다시 검사해야 한다**

---

## 3. HTTP & JSON (실전에서 이렇게 쓰인다)

### 3-1. HTTP = 통신 규칙

|메서드|의미|예시|
|---|---|---|
|GET|조회|팀원 목록|
|POST|생성|모집글 등록|
|PATCH|수정|프로필 수정|
|DELETE|삭제|게시글 삭제|

---

### 3-2. JSON = 데이터 포맷 (언어 아님)

```json
{
  "title": "스터디 모집",
  "headcount": 4,
  "part": "BACKEND"
}
```

- JSON은 **자연어 표현이 가능한 데이터 표기법**
    
- 실행되지 않는다
    
- 컴파일 대상도 아니다
    
- **파싱해서 값만 꺼내 쓴다**
    

---

## 4. DTO / Entity / Repository의 정확한 역할

### 4-1. DTO (Data Transfer Object)

> **“계층 사이를 오가는 데이터 그릇”**

#### Request DTO

- 프론트 → 백 요청용
    
- JSON → Java 객체로 변환된 결과
    

```java
public record RecruitCreateRequest(
    String title,
    String description,
    Integer headcount,
    Part part
) {}
```

#### Response DTO

- 백 → 프론트 응답용
    
- Entity를 그대로 보내지 않는다
    

```java
public record RecruitResponse(
    Long id,
    String title,
    Integer headcount,
    String status
) {}
```

---

### 4-2. Entity (DB 모델)

> **“DB 테이블과 1:1에 가까운 서버 내부 모델”**

```java
@Entity
public class Recruit {
    @Id @GeneratedValue
    private Long id;

    private String title;
    private Integer headcount;

    @Enumerated(EnumType.STRING)
    private Part part;
}
```

⚠️ Entity는 **절대 API 응답으로 직접 반환하지 않는다**

---

### 4-3. Repository

> **“Entity를 DB에 저장/조회하는 계층”**

```java
public interface RecruitRepository
    extends JpaRepository<Recruit, Long> {
}
```

- SQL 직접 작성 ❌
    
- Entity 단위로 저장/조회 ⭕
    

---

## 5. JSON → DTO → Service → Entity → DB → Response DTO

### 5-1. 요청 처리 흐름 (실제 동작)

1️⃣ 프론트에서 JSON 전송  
2️⃣ Spring이 `@RequestBody`로 받음  
3️⃣ **Jackson**이 JSON → Request DTO로 역직렬화  
4️⃣ Controller → Service 호출  
5️⃣ Service에서 Entity 생성/수정  
6️⃣ Repository로 DB 처리  
7️⃣ 결과를 Response DTO로 변환  
8️⃣ Jackson이 Response DTO → JSON 직렬화  
9️⃣ HTTP 응답 전송

---

## 6. Jackson & 역직렬화의 정확한 의미

- Jackson은 JSON의 key와 DTO의 필드명을 **매칭**
    
- 메서드 선택과는 무관
    
- URL + HTTP Method로 컨트롤러 결정 후 DTO 매핑
    

---

## 7. 객체 / 인스턴스 / 필드 / 프로퍼티 정리

|용어|의미|
|---|---|
|클래스|설계도|
|인스턴스|메모리에 생성된 실제 객체|
|객체|인스턴스와 거의 동일한 의미|
|필드|클래스 내부 변수|
|프로퍼티|외부에서 접근 가능한 논리적 속성|
|컬럼|DB 테이블의 열|

📌 DTO 하나 = **객체/인스턴스**  
📌 DTO 안의 `name`, `age` = **필드**

## 8. Enum (실전에서 반드시 써야 하는 이유)

### 8-1. 왜 Enum인가

- 문자열 오타 방지
    
- 프론트/백/DB 값 통일
    
- 검색/필터/통계 안정성
    

```java
public enum Part {
    BACKEND,
    FRONTEND,
    PM,
    DESIGN
}
```

- 한글 Enum ❌ (비추천)
    
- 화면 표시만 한글 변환 ⭕
    

---

## 9. OAuth + JWT 인증 (실전 흐름)

### 9-1. 왜 서버가 인증을 책임져야 하나

- 프론트 코드는 조작 가능
    
- 이메일 도메인 검사도 서버가 최종 판단
    

---

### 9-2. Google OAuth 실전 흐름

1️⃣ 사용자가 Google 로그인  
2️⃣ Google → 서버로 인증 결과 전달  
3️⃣ 서버가 Google API로 **직접 사용자 정보 조회**  
4️⃣ `@catholic.ac.kr` 도메인 검증  
5️⃣ 통과 시 JWT 발급  
6️⃣ 이후 요청은 JWT 기반 인증

---

## 10. 토큰 개념 정리 (졸업 프로젝트 기준)

### 10-1. Access Token

- API 요청 시 인증용
    
- JWT 형식
    
- 유효기간 짧게
    

### 10-2. Refresh Token (선택)

- Access 만료 시 재발급용
    
- 구현 난이도 ↑
    
- 졸업 프로젝트에서는 **Access Token만으로도 충분**
    

---

## 11. 토큰 저장 위치 (프론트 협의 필수)

### localStorage

- 구현 쉬움
    
- XSS 위험
    

### HttpOnly Cookie

- JS 접근 불가 → 토큰 탈취 방어
    
- CSRF 대응 필요
    
- 실무에서는 선호됨

## 12. 에러 응답 포맷 (무조건 고정)

```json
{
  "errorCode": "AUTH_401",
  "message": "로그인이 필요합니다"
}
```

- `errorCode`: 프론트 분기용 (고정)
    
- `message`: 사용자 안내

## 14. 컴파일러 / JVM / JIT 정리 (정확 버전)

- `.java → .class` : **컴파일러(javac)**
    
- 실행 시:
    
    - JVM이 바이트코드 해석 실행(인터프리터)
        
    - 자주 쓰는 코드 → JIT로 기계어 변환 + 캐싱
        

👉 Java = **컴파일 + 인터프리트 + JIT 혼합 모델**
