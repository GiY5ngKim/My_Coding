# 코드메이트 프로젝트

## ✅ 백엔드가 해야 하는 작업 전체 로드맵

---

## 0단계. “백엔드의 책임 범위”를 먼저 고정

> 이걸 안 하면 중간에 계속 일이 튄다.

백엔드 책임:

- **데이터의 진실(Source of Truth)**
    
- **API 계약**
    
- **보안/검증**
    
- **데이터 무결성**
    

프론트 책임:

- 화면/UX
    
- API 호출
    
- 사용자 경험
    

👉 이 인식이 없으면  
“이건 프론트에서 하면 되지 않나?” → 나중에 백이 다 뒤집어씀.

---

## 1단계. 요구사항 해석 & MVP 범위 고정

**(코딩 ❌, 사고 단계 ⭕)**

### 해야 할 것

- 기능 목록을 **MVP 기준으로 자른다**
    
    - 로그인/회원가입
        
    - 프로필(part, bio, skills)
        
    - 팀원 검색
        
    - 프로젝트 모집
        
    - 지원/찜
        
- “이번에 안 할 것”을 명확히 적는다
    
    - 코드리뷰 고도화
        
    - AI 추천 고급화
        
    - 알림 시스템 등
        

### 왜 필요?

- 백엔드는 **나중에 추가되는 기능 때문에 현재 설계를 깨먹는 역할**을 맡기 쉬움
    
- “지금 안 할 것”을 적어두면 방어가 된다
    

---

## 2단계. 핵심 도메인 정의 (개념 모델링)

**(아직 DB 아님)**

### 해야 할 것

- 이 서비스에 **실제로 존재하는 개념**만 뽑기
    

예:

- User
    
- Profile
    
- Skill
    
- Project
    
- ProjectRole
    
- Application
    
- Bookmark
    

### 여기서 반드시 결정해야 하는 질문

- User ↔ Profile은 1:1인가?
    
- Project는 Role을 여러 개 가질 수 있는가?
    
- Skill은 공용 테이블인가?
    

👉 이 단계에서 헷갈리면 **DB 설계가 계속 흔들림**

---

## 3단계. Enum / 고정 값 설계

**(오늘 대화의 핵심 지점)**

### 반드시 고정해야 하는 값

- `part` (FRONTEND / BACKEND / PM / DESIGN / ETC)
    
- `projectStatus` (OPEN / CLOSED)
    
- `applicationStatus` (PENDING / ACCEPTED / REJECTED)
    

### 왜 지금?

- 나중에 바꾸면 **DB 마이그레이션 + API 계약 변경**이 생김
    
- 지금은 비용이 가장 싸다
    

---

## 4단계. ERD 설계 (DB 구조 확정)

**(백엔드가 주도해야 하는 핵심 단계)**

### 해야 할 것

- 테이블
    
- PK / FK
    
- 관계(1:1, 1:N, N:M)
    
- 유니크 제약조건
    

예:

- `profiles.user_id UNIQUE`
    
- `applications (user_id, project_id) UNIQUE`
    

### 산출물

- ERD 이미지 or 문서
    
- 컬럼 정의 표
    

👉 **이걸 기준으로 프론트/백이 동시에 움직이기 시작함**

---

## 5단계. DB 연동 세팅

(이전에 설명한 SQL 단계)

### 해야 할 것

- DB 선택 (Postgres/MySQL)
    
- Docker or 로컬 설치
    
- `application.yml` 작성
    
- JPA 설정
    
- 연결 확인
    

👉 이 단계가 끝나야 **“서버가 살아있다”**고 말할 수 있음

---

## 6단계. Entity 작성

**(ERD → 코드로 변환)**

### 해야 할 것

- 테이블당 Entity 1개
    
- 관계 매핑
    
- Enum 적용
    
- 제약조건 어노테이션
    

### 주의

- Entity는 **저장용**
    
- 화면용 데이터 넣지 말 것
    

👉 여기서부터 **DB 구조 변경 비용이 생기기 시작함**

---

## 7단계. Repository 작성

**(DB 접근 통로)**

### 해야 할 것

- 기본 CRUD
    
- 검색에 필요한 메서드
    
- 중복 체크용 exists 쿼리
    

👉 아직 로직 없음  
👉 “DB랑 대화만 가능”한 상태

---

## 8단계. API 설계 & 명세 작성 ⭐⭐⭐

**(프론트가 움직이느냐 마느냐가 여기서 결정됨)**

### 해야 할 것

- URL
    
- Method
    
- Request DTO
    
- Response DTO
    
- 상태코드
    

### 산출물

- Swagger(OpenAPI)
    

👉 **백엔드는 이걸로 프론트에게 ‘계약’을 준다**

---

## 9단계. DTO 설계

**(프론트와의 실제 데이터 계약)**

### 해야 할 것

- Request DTO / Response DTO 분리
    
- Entity → DTO 변환 기준 정의
    
- 불필요한 필드 차단
    

👉 여기서 보안 사고 많이 난다 (Entity 그대로 반환 ❌)

---

## 10단계. Controller 작성

**(HTTP 입구)**

### 해야 할 것

- URL 매핑
    
- 인증 여부 체크
    
- Validation
    
- Service 호출
    

### 원칙

- 로직 ❌
    
- 분기 ❌
    
- “연결만 한다”
    

---

## 11단계. Service 작성

**(백엔드의 핵심 노동 구간)**

### 해야 할 것

- 비즈니스 규칙
    
- 상태 전이
    
- 중복 방지
    
- 트랜잭션 처리
    
- Enum 기반 분기
    

👉 여기서 “백엔드가 죽는” 일이 생긴다  
(데이터 + 규칙 + 예외가 몰림)

---

## 12단계. 인증 / 인가 구현

**(JWT 등)**

### 해야 할 것

- 로그인
    
- 토큰 발급
    
- 사용자 식별
    
- 권한 체크
    

👉 이걸 뒤로 미루면  
나중에 Controller/Service를 다 고친다

---

## 13단계. 예외 처리 & 에러 규약

**(운영 안정성)**

### 해야 할 것

- 공통 에러 응답 형식
    
- 예외 매핑 (400 / 401 / 403 / 404 / 409)
    
- 의미 있는 메시지
    

👉 “왜 안 되는지”를 프론트가 알 수 있어야 함

---

## 14단계. 프론트 연동 & 계약 조정

**(현실과 이상이 충돌하는 구간)**

### 해야 할 것

- 프론트 요청 반영
    
- DTO 수정
    
- API 명세 보정
    
- 하위호환 유지 판단
    

👉 여기서 백엔드 체력 많이 소모됨

---

## 15단계. 테스트 & 데이터 검증

**(안 하면 반드시 터짐)**

### 해야 할 것

- 핵심 API 수동 테스트
    
- 중복/경계 조건 테스트
    
- Enum 변경 시 영향 확인
    

---

## 16단계. 배포 준비

**(로컬 → 운영)**

### 해야 할 것

- DB 환경 분리
    
- 설정 값 분리
    
- ddl-auto 조정
    
- 마이그레이션 전략 고민
    

---

## 17단계. 배포 & 운영 대응

**(진짜 실무)**

### 해야 할 것

- 장애 로그 확인
    
- 데이터 수정 요청 대응
    
- 기능 추가 요청 검토
    

👉 이 단계에서  
“아 왜 처음에 이렇게 설계했지…”가 나온다  
→ 그래서 앞단계가 중요함

---

## 🔥 핵심 요약 (백엔드 관점)

- 백엔드는 **기능 만드는 사람**이 아니라  
    **데이터와 규칙을 책임지는 사람**
    
- “기능 하나 추가”는  
    코드 문제가 아니라 **데이터/계약/운영 문제**


# 핵심 규칙

1. Controller는 DB 직접 접근 금지
    
2. Controller는 Service만 호출
    
3. Service가 비즈니스 규칙 담당
    
4. Repository는 DB 접근만 담당
    
5. DTO는 api/**/dto
    
6. Entity는 domain/**
    
7. Entity를 API 응답으로 직접 반환 금지
    
8. domain은 api를 import 금지
    
9. 공통 예외는 GlobalExceptionHandler에서 통일
    
10. Enum/상태값은 domain/common에 둔다
