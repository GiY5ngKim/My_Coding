## STEP 1. Spring Initializr에서 프로젝트 생성(웹)

### 왜 웹 Initializr로 시작하나

- **표준 템플릿을 자동 생성**해줘서, 빌드/실행/폴더 구조가 처음부터 안정적입니다.
    
- Gradle Wrapper(`gradlew`)까지 포함되기 때문에, **개발 PC가 달라도 동일한 Gradle 버전**으로 빌드가 됩니다(협업/재현성 핵심).
    

### 입력값

- Project: Gradle / Language: Java / Boot: 3.x / Jar / Java 17
    
- Group: `com.cat`
    
- Artifact/Name: `picshare`
    
- Package name: `com.cat.picshare`
    
- Dependencies: Spring Web, Spring Data JPA, Validation, Spring Security, Lombok, H2 Database
    

### 왜 이 조합인가(요구사항 매핑)

- Spring Web: REST API
    
- Spring Data JPA: ERD 기반 DB 접근
    
- Validation: 요청 검증 표준화(에러 통일과 직결)
    
- Spring Security: JWT 인증/인가 기반
    
- Lombok: DTO/엔티티 보일러플레이트 감소
    
- H2 Database : 로컬에서 DB 설치 없이 빠르게 검증

## STEP 2. ZIP 압축 해제 후 IntelliJ로 열기

### 왜 “압축 해제 → IntelliJ Open”인가

- IDE가 Gradle 프로젝트로 인식하면서 **의존성 다운로드/빌드 설정**을 자동으로 맞춥니다.
    
- “프로젝트 루트”가 명확해져서 이후 스크립트/배포 구조도 동일하게 유지됩니다.
    

### 반드시 확인(왜 중요?)

- Project SDK = 17, Gradle JVM = 17
    
    - 이유: Java 버전이 엇갈리면 **컴파일/런타임 불일치**로 빌드 실패가 쉽게 납니다.

## STEP 3. main 실행(서버 기동)

### 왜 바로 실행부터 하냐

- 서버가 안 켜지면 이후 설정/코드 작업은 전부 무의미합니다.
    
- “환경 문제인지, 코드 문제인지”를 분리하기 위해 **가장 먼저 ‘기동 성공’**을 확인합니다.
    
## 성공했을 때 반드시 확인해야 할 로그

아래가 **나오면 성공**이다.

- `Starting PicshareApplication`
    
- `Tomcat started on port(s): 8080`
    
- `Started PicshareApplication in X seconds`
    
이 3줄 중 **하나라도 안 나오면 실패**다.

### 성공 기준(왜 Whitelabel이어도 OK?)

- 컨트롤러가 아직 없으면 404/Whitelabel이 정상입니다.
    
- 중요한 건 “Spring 컨텍스트가 올라오고 Tomcat이 포트를 리슨한다”는 사실입니다.
    


### 브라우저 확인

http://localhost:8080

- Whitelabel Error Page 또는 404 응답 → **정상**
    
- 이 시점에는 컨트롤러가 없기 때문에 에러 화면이 정상 동작이다.

## Spring Security 기본 로그인 화면 확인(초기 상태)

### 현상

- `/` 접속 시 로그인 화면이 표시됨
    

### 원인

- Spring Security 의존성을 추가하면, 기본 보안 설정이 자동 적용되어
    
    - 모든 요청에 인증 요구
        
    - 인증되지 않은 경우 기본 로그인 페이지 제공
        

### 판단

- 이는 오류가 아니며, **Spring Security가 정상 동작하고 있다는 증거**다.
    
- 단, API 서버 목적에는 맞지 않으므로 이후 제거 대상이다.
