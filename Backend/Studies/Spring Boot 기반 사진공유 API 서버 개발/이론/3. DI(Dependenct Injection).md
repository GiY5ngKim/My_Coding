##### DI : 객체를 내가 직접 만들지 않고 외부(Spring)가 만들어서 넣어주는 것
```
# DI 미사용 방식

public class UserController {
    private UserService userService = new UserService();
```
	* 생성 책임 : 컨트롤러 / 결합도 : 최악  →  사용하면 안됌
```
# DI 방식 (Spring 방식)

@RestController
public class UserController {

    private final UserService userService;

    public UserController(UserService userService) {
        this.userService = userService;
    }
}
```
	* new UserService() 가 없으므로 Spring이 UserService를 만들어서 넣어줌

##### Spring은 어떻게 이게 가능한가?
Spring은 실행 시 다음과 같은 작업을 수행한다.
1. @Component, @Service, @Controller 같은 클래스 스캔
2. 객체 생성
3. 객체 간 의존 관계 파악
4. 필요한 곳에 자동으로 주입
	→이걸 관리하는 공간이 Spring Container(loC Container)

##### DI의 목적은 역할 분리, 결합도 감소, 테스트 가능성 확보
* Controller : 요청/응답
* Service : 비즈니스 로직
* Repository : DB 접근
	→ 서로 new 생성 X, 전부 Spring이 연결

##### @Service : 이 클래스는 비즈니스 로직을 담당하는 Spring 관리 객체이다
```
@Service
public class MessageService {
}
```
	이 순간 Spring은 다음과 같은 작업을 수행한다.
	1. 애플리케이션 시작 시 이 클래스를 스캔
	2. 객체(Bean)를 한 번 생성
		- Bean : Spring이 생성하고 관리하는 객체
			→ 생성 시점, 생명 주기, 의존 관계, 프록시를 모두 Spring이 관리
	3. Spring 컨테이너에 등록
	4. 필요한 곳에 주입 가능한 상태로 유지


##### DI 방식
1. 필드 주입(금지)
```
@RestController
public class MessageController {

    @Autowired
    private MessageService messageService;
}
```
	* 테스트 불가(생성자 없음)
	* final 불가
	* 객체 생성 시점이 명확하지 않음

2. 세터 주입(가끔)
```
@RestController
public class MessageController {

    private MessageService messageService;

    @Autowired
    public void setMessageService(MessageService messageService) {
        this.messageService = messageService;
    }
}
```
	* 객체 생성 후 주입
	* 선택적 의존성에 사용 가능
	* 불변성 깨짐
	* 누락 위험

3. 생성자 주입(표준)
```
@RestController
public class MessageController {

    private final MessageService messageService;

    public MessageController(MessageService messageService) {
        this.messageService = messageService;
    }
}
```
	* 객체 생성 시점에 의존성 확정
	* final 가능
	* 테스트 쉬움
	* 순환 참조 즉시 발견
	* Spring 공식 권장

##### Spring은 생성자 주입을 어떻게 알아서 해주나?
- 컨트롤러가 생성자 하나 뿐이면 @Autowired 생략 가능
1. Spring이 내부적으로 MessageController 생성자 확인
2. 파라미터 타입(MessageService) 탐색
3. 해당 타입의 Bean 찾기
4. 주입

##### 메신저 구현 Service, Controller 코드
```
#MessageService

// 이 클래스가 속한 패키지 경로를 선언
// → 자바 네임스페이스, Spring의 컴포넌트 스캔 기준
package com.example.messenger.service;

// Spring에게 "이 클래스는 비즈니스 로직 담당 객체다"라고 알려주는 어노테이션
// → 애플리케이션 시작 시 Spring이 이 클래스를 스캔해서 객체(Bean)로 생성함
import org.springframework.stereotype.Service;

// @Service는 @Component의 특수화 버전
// → 의미적으로 "비즈니스 로직 담당"이라는 의도를 드러냄
// @ 없으면 MessageService가 Bean으로 등록되지 않음
	→ 컨트롤러 생성 시 주입 실패 → 애플리케이션 시작 자체가 에러로 종료
@Service
public class MessageService {

    // 메시지를 전송하는 비즈니스 로직 메서드
    // → HTTP, 컨트롤러, 요청 방식과 전혀 무관
    // → 순수하게 "메시지를 보낸다"라는 도메인 행위만 표현
    public void send(String content) {

        // 지금은 단순히 콘솔 출력
        // → 나중에는 여기서 DB 저장, 검증, WebSocket 전송 등으로 확장됨
        System.out.println("메시지 전송: " + content);
    }
}
```
- Service 관점 : HTTP, URL, JSON 전혀 모름, 비즈니스 행위만 알고 있음

```
#MessageController

// 이 컨트롤러 클래스가 속한 패키지
// → 메인 패키지(com.example.messenger) 하위라서 Spring이 자동 스캔 가능
package com.example.messenger.api;

// MessageService 타입을 사용하기 위해 import
// → 직접 new 하지 않고, 생성자 주입으로 받기 위한 준비
import com.example.messenger.service.MessageService;

// HTTP 관련 어노테이션 import
import org.springframework.web.bind.annotation.PostMapping;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;

// @RestController = @Controller + @ResponseBody
// → 이 클래스는 HTTP 요청을 처리하고
// → 메서드 반환값을 HTTP 응답 바디로 바로 사용한다
@RestController
public class MessageController {

    // 컨트롤러가 의존하는 서비스
    // final: 생성자에서 반드시 초기화되어야 하며, 이후 변경 불가
    // → 컨트롤러는 항상 "정상적인 Service"를 가진 상태로 존재함
    private final MessageService messageService;

    // 생성자
    // → Spring이 이 생성자를 보고 MessageService Bean을 찾아서 주입함
    // → new MessageService()를 우리가 직접 쓰지 않음 (DI 핵심)
    public MessageController(MessageService messageService) {
        this.messageService = messageService;
    }

    // HTTP POST 요청 중,
    // URL 경로가 "/messages" 인 요청이 오면 이 메서드를 실행하라는 선언
    // @ 없으면 요청, 메서드 연결 X → 서버는 켜지지만 기능 없음 (404)
    @PostMapping("/messages")
    public void send(

            // HTTP 요청 파라미터 중 "content" 값을 읽어서
            // String content 변수에 넣어달라는 의미
            // 예: POST /messages?content=hello
            //@ 없으면 content 값을 HTTP 요청에서 꺼내올 수 없음
               → 파라미터 바인딩 실패
            @RequestParam String content
    ) {

        // 컨트롤러는 "일을 직접 하지 않는다"
        // → 요청을 해석해서 서비스로 넘겨줄 뿐
        messageService.send(content);
    }
}
```
