#### 필수 개발 도구

1. JDK (Java 17)
    - 역할
        - Spring Boot 3.x 애플리케이션의 컴파일 및 실행을 담당하는 런타임 환경
        - Gradle 빌드, 테스트, 애플리케이션 실행에 사용되는 JVM
            
    - 왜 17인가
        - Spring Boot 3.x의 최소 요구 및 권장 LTS(Long Term Support) 버전
        - 장기 지원으로 운영 환경에서 안정적인 유지보수 가능
        - 개발/운영 환경의 JVM 버전 통일이 용이
            
    - 체크 포인트
        - IntelliJ Project SDK = Java 17
        - Gradle JVM = Java 17
        - `java -version`으로 로컬 설치 버전 확인
            
2. IDE (IntelliJ IDEA)
    - 역할
        - Spring Boot + Gradle 프로젝트의 생성, 실행, 디버깅 환경 제공
        - 코드 자동 완성, 패키지 구조 관리, 리팩토링 지원
        - 테스트 코드 실행 및 디버깅에 유리
            
    - 왜 IntelliJ인가
        - Spring Boot, JPA, Gradle과의 통합 지원이 가장 안정적
        - Run/Debug 설정, Profile 전환, 로그 확인이 직관적
        - 실무에서 가장 보편적으로 사용되는 Java 백엔드 IDE
    
3. Git
    - 역할    
        - 소스 코드 버전 관리 및 변경 이력 추적
        - 브랜치 기반 개발 및 협업 흐름 지원
    
    - 필수 이유    
        - 학습 과정에서 변경 사항을 안전하게 되돌릴 수 있음        
        - PR, 코드 리뷰, 이슈 단위 개발 흐름을 그대로 연습 가능
        - 배포 및 운영 단계에서 변경 이력 추적에 필수
            
4. DB (로컬: H2 권장, 운영 대비: MySQL/PostgreSQL)
    - 역할
        - JPA 엔티티, 리포지토리, 트랜잭션 동작 검증
        - 데이터 모델(ERD)과 실제 DB 구조의 일치 여부 확인
    
    - 왜 H2로 시작하나    
        - 별도 설치 없이 빠른 로컬 실행 가능
        - 메모리/파일 모드 지원으로 개발 편의성 높음
        - ERD 설계를 JPA 기반으로 빠르게 검증 가능
    
    - 운영 DB를 병행하는 이유(선택)
        - H2와 MySQL/PostgreSQL 간 차이(DDL, 타입, 인덱스, 제약 조건)를 사전에 경험
        - 운영 환경 이슈를 개발 단계에서 최소화
    
5. API 테스트 도구 (Postman / Insomnia / curl)
    - 역할
        - 프론트엔드 없이 API 요청/응답 테스트
        - 회원가입, 로그인, 토큰 인증(JWT) 시나리오 검증
        - 에러 응답 및 권한 처리 테스트

#### Spring Boot 프로젝트 생성 및 기본 구조 (Gradle + Spring Boot)

프로젝트 생성 기준 (Spring Initializr 권장)  
1. Build Tool: Gradle  
2. Language: Java  
3. Spring Boot: 3.x  
4. Packaging: Jar  
5. Java: 17

의존성  
1. Spring Web  
* 역할  
	- REST API 구현(HTTP 요청/응답 처리)  
	- 객체 ↔ JSON 자동 직렬화/역직렬화  
	* 요구사항 대응  
	- “응답 포맷 통일(JSON)”의 기반 제공

 2. Spring Data JPA
	* 역할
		- DB 접근 계층(Repository) 제공
		- 트랜잭션 관리 및 엔티티 기반 데이터 처리
	* 요구사항 대응
		- “DB 트랜잭션(서비스 계층)”
		- “Auditing(createdAt, updatedAt)”

3. Validation
	* 역할
		- 요청 DTO의 유효성 검증(@NotNull, @Email, @Size 등)
	* 요구사항 대응
		- 입력 검증 표준화
		- 예외 처리/응답 통일과 연계

4. Spring Security
	* 역할
		- 인증/인가 처리
		- Access Token(JWT) 기반 요청 접근 제어
	* 요구사항 대응
		- “보안(비밀번호 해시, JWT)”

5. Lombok
	* 역할
		- Getter/Setter, 생성자, Builder 등 보일러플레이트 코드 제거
		- 엔티티/DTO 가독성 및 생산성 향상

6. DB Driver
	* H2 (로컬 개발용)
	* MySQL / PostgreSQL (운영 환경 대비)

7. (선택) Actuator
	* 역할
		- 서버 상태 및 헬스 체크 엔드포인트 제공
		- 배포 후 서버 정상 동작 여부 확인에 활용

8. (선택) AWS SDK S3
	* 역할
		- Presigned URL 발급
		- 파일 업로드 기능 구현 시 사용

#### 애플리케이션 설정 (application.yml) 설계
- 목표
    - 로컬 환경에서 즉시 실행 가능한 최소 설정 구성
    - 운영 환경(prod)은 프로필 분리를 통해 관리
        
- 프로필 분리(권장)
    1. application.yml (공통)
        - 서버 공통 설정
        - 포트
        - 기본 로깅 정책
        - 공통 설정 값
            
    2. application-local.yml (로컬)
        - 로컬 DB(H2/MySQL) 접속 정보
        - 개발 편의용 JPA 설정
        - SQL 로그 출력
            
    3. application-prod.yml (운영)
        - 운영 DB 접속 정보
        - 보안 강화 설정
        - 로그 레벨 최소화
        - 민감 정보는 환경 변수로 주입
            
- 필수 설정 항목(보편)
1. server.port        
        - 역할
            - 애플리케이션 실행 포트 지정(예: 8080)

2. spring.datasource
	* 역할
		- DB 연결 정보 설정
		- URL / Username / Password / Driver

3. spring.jpa
	* 역할
		- JPA 동작 방식 제어
		- ddl-auto, show-sql, dialect 등

4. logging.level
	* 역할
		- 패키지/환경별 로그 레벨 제어
		- 개발/운영 환경 로그 분리

5. jwt 설정 (추후 Security 구현에 사용)
	* 예시 항목
		- secret
		- access-token-expiration
		- refresh-token-expiration (선택)
