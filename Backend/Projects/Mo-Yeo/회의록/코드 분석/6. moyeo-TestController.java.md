## ✅ 줄별 역할/기능 해설

```java
package com.catholic.moyeo;
// 이 클래스가 속한 패키지 선언.
// 현재 루트 패키지 바로 아래에 위치하고 있음.

import org.springframework.stereotype.Controller;
// Spring MVC 컨트롤러 선언용 어노테이션.
// 기본적으로는 View(HTML 템플릿) 반환을 전제로 하는 컨트롤러.

import org.springframework.web.bind.annotation.GetMapping;
// HTTP GET 요청을 특정 메서드에 매핑하기 위한 어노테이션.

import org.springframework.web.bind.annotation.ResponseBody;
// 메서드 반환 값을 View가 아니라 HTTP 응답 Body에 직접 쓰도록 지정하는 어노테이션.

@Controller
// 이 클래스를 Spring MVC 컨트롤러로 등록.
// @RestController가 아닌 @Controller이므로 기본은 View 반환 구조.
public class TestController {

    @GetMapping("/oauth/callback")
    // HTTP GET /oauth/callback 요청이 들어오면
    // 아래 callback() 메서드가 실행된다.
    // 일반적으로 OAuth 리다이렉트 콜백 URL로 사용하는 경로.

    @ResponseBody
    // 이 메서드의 반환 값을 템플릿으로 렌더링하지 않고
    // HTTP 응답 Body에 그대로 작성하도록 지정.
    // (문자열이면 그대로 문자열 응답)

    public String callback() {
        // OAuth 콜백이 들어왔을 때 실행되는 메서드.
        // 현재는 실제 인증 처리 로직은 없고 단순 테스트용 응답만 반환.

        return "구글 로그인 성공!";
        // 클라이언트에게 단순 문자열 응답.
        // JSON이 아니라 text/plain 형태로 반환된다.
    }
}
```

# 🔴 P0 — 반드시 정리 필요 (구조 충돌 가능성)

---

## 1️⃣ OAuth2 로그인 구조와 충돌 가능성

현재 WebSecurityConfig에:

```java
.oauth2Login(...)
```

그리고 permitAll에:

```
"/oauth/callback"
```

이 존재.

즉:

- Spring Security OAuth2 리다이렉트 플로우를 쓰는 구조일 가능성 있음.
    
- 그런데 현재 이 컨트롤러는 단순 문자열 반환.
    

⚠️ 만약 oauth2Login을 실제로 쓰고 있다면:

- 이 컨트롤러는 필요 없음.
    
- Spring Security가 내부적으로 callback 처리.
    

⚠️ 만약 idToken 방식만 쓸 거라면:

- 이 컨트롤러는 완전히 불필요.
    

👉 구조 혼합 상태일 가능성 있음.

---

## 2️⃣ 현재 기능적 의미 없음

이 컨트롤러는:

- 로그인 성공 메시지만 출력
    
- 토큰 발급 없음
    
- 세션 처리 없음
    
- 리다이렉트 없음
    

실서비스에서는 의미 없는 테스트 코드.

# 🟡 P1 — 실무 권장

---

## 3️⃣ @RestController로 통일

현재:

```java
@Controller
@ResponseBody
```

대신

```java
@RestController
```

로 통일 가능.

---

## 4️⃣ 위치 분리

현재 위치:

```
com.catholic.moyeo
```

실무 권장:

```
com.catholic.moyeo.auth.controller
```

또는

```
com.catholic.moyeo.oauth.controller
```
