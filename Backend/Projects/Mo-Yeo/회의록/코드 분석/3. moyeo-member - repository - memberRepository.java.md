## ✅ 줄별 역할/기능 해설

```java
package com.catholic.moyeo.member.repository;
// 패키지 선언. Member 관련 Repository 계층을 모아둔 경로.

import com.catholic.moyeo.member.domain.Member;
// 이 Repository가 관리할 엔티티 타입(Member)를 임포트.

import org.springframework.data.jpa.repository.JpaRepository;
// Spring Data JPA가 제공하는 기본 CRUD/조회 기능 인터페이스.

import java.util.Optional;
// 조회 결과가 없을 수 있음을 표현하기 위한 Optional 타입.

public interface MemberRepository extends JpaRepository<Member, Long> {
// Member 엔티티에 대한 Repository 인터페이스.
// JpaRepository<Member, Long> 상속으로 기본 기능 제공:
// - save, findById, findAll, deleteById, existsById 등
// 두 번째 제네릭(Long)은 Member의 PK 타입.

    Optional<Member> findByProviderAndProviderSub(String provider, String providerSub);
    // 메서드 이름 기반 쿼리 생성(Query Method).
    // provider와 providerSub가 일치하는 회원을 조회.
    // (provider, provider_sub) 유니크 제약이 있으므로 0 또는 1개만 나오는 게 정상.
    // 결과가 없을 수 있으니 Optional로 감싼다.

    Optional<Member> findByEmail(String email);
    // 이메일로 회원 조회.
    // email 컬럼이 유니크 제약이므로 0 또는 1개만 나오는 게 정상.
}
```

## 🔴 P0 — 반드시 수정/점검 (실제 장애/보안/정합성 리스크)

### 1️⃣ `provider`를 String으로 조회하는 구조(오타/정합성 리스크)

#### 📌 현재 코드

```java
Optional<Member> findByProviderAndProviderSub(String provider, String providerSub);
```

#### ❌ 문제 이유

- 서비스/핸들러에서 `"google"`, `"Google"` 같은 오타/대소문자 문제로 조회 실패 가능
    
- 조회 실패가 곧 “회원 미존재로 판단 → 중복 생성 시도 → 유니크 제약 위반” 같은 장애로 이어질 수 있음
    

#### ✅ 수정안(실무 추천)

- `provider`를 Enum으로 강제하고 Repository 시그니처도 Enum 사용
    

```java
Optional<Member> findByProviderAndProviderSub(AuthProvider provider, String providerSub);
```

> 단, 이건 **Member 엔티티의 provider 타입 변경(@Enumerated)**과 같이 가야 함.

---

### 2️⃣ “중복 가입” 동시성 시나리오 대비(서비스 레벨 필수)

#### 📌 현재 Repository 자체는 문제 없음

하지만 실무에서 장애가 나는 지점은 보통:

- 동시에 두 요청이 들어와서
    
- 둘 다 “없네?” 판단 후 save 시도
    
- DB에서 unique constraint 위반 발생
    

#### ✅ 실무 권장 처리

- 서비스에서 save 시 **DataIntegrityViolationException** 잡아서 “이미 가입됨” 흐름으로 처리
    
- 또는 “조회 + 저장”을 트랜잭션으로 묶고 재시도 정책
    

> Repository 수정이 아니라 **서비스 로직 보완**이 P0.

## 🟡 P1 — 실무 권장 (졸업프로젝트에서는 선택)

### 3️⃣ `findByEmail` 사용처 정책 명확화

#### 📌 현재 코드

```java
Optional<Member> findByEmail(String email);
```

#### 실무 관점

- OAuth 기준 사용자 식별은 원칙적으로 `(provider, provider_sub)`가 정석
    
- email은 변경 가능/정책에 따라 중복 가능
    

#### 권장

- “회원 식별”은 `(provider, providerSub)`로 하고
    
- email 조회는:
    
    - 관리자 기능
        
    - 사용자 검색
        
    - 이메일 중복 방지 체크  
        같은 용도로만 제한하는 게 정합성 측면에서 안전
        

> 다만 너희는 email도 unique로 걸어놨으니, 지금 단계에서는 사용해도 큰 문제는 없음.

---

### 4️⃣ 성능/인덱스: 유니크 제약이면 충분

- DB에서 unique constraint는 인덱스로 동작하는 경우가 대부분이라  
    조회 성능은 보통 OK
    
- 별도 인덱스 어노테이션 추가는 우선순위 낮음

## ✅ 최종 정리

- Repository 구조 자체는 깔끔하고 정상.
    
- **실무 리스크는 “provider String 오타” + “동시성 중복 생성”**이 핵심.
    
- 개선 우선순위:
    
    1. provider Enum 강제(가능하면)
        
    2. 서비스 레벨에서 unique 위반 처리/재시도 설계
