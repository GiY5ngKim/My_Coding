## 2) Auth / Security (중요)

### 인증 흐름(큰 그림)

- OAuth2 로그인 성공
    
- 서버가 JWT 발급
    
- JWT는 **HttpOnly Cookie**로 내려가고, 이후 요청은 쿠키로 인증됨 (프론트에서 Authorization 헤더 직접 세팅 최소화)
    
- SecurityContext principal에는 **user_id(Long 또는 String)** 가 들어간다고 가정
    

### 현재 인증 주체 ID 꺼내는 방식

`AuthUtil.currentMemberId()` 사용 (중요: 전 모듈 통일)

- `SecurityContextHolder.getContext().getAuthentication().getPrincipal()`
    
- principal 타입이 `Long`이면 그대로, `String`이면 `Long.parseLong`
    

⚠️ 주의:

- principal이 다른 타입(CustomUser 등)으로 바뀌면 AuthUtil도 같이 바뀌어야 함.
    
- 그래서 “principal=Long” 설계를 팀 전원이 공유해야 함.

## 4) Recruit 모듈 정책(핵심)

### 엔티티/테이블 개념

- `recruit_post` : 모집글
    
- `recruit_application` : 지원 row
    

### 모집글 상태 정책

- `OPEN / CLOSED`
    
- deadline이 지나면 **자동 CLOSED 전환(스케줄러/배치)**로 가는 방향(선택지 A)
    
- 동시에 서비스에서도 “행위 시점 검증”은 유지(안전장치)
    

### 지원 정책(가장 중요)

- 지원/취소는 **멱등(idempotent)**
    
    - 이미 지원한 상태에서 apply 재호출 → 200 OK + 동일 상태
        
    - 지원 안 한 상태에서 cancel 재호출 → 200 OK + 동일 상태
        
- 취소는 **row 삭제**
    
    - CANCELLED 상태 없음
        
- CLOSED or deadline 지난 경우:
    
    - apply/cancel/decide 모두 불가
        
    - 정책상 “행위 불가”는 400(IllegalStateException)로 통일
        

### 카운트 정책(applicant_count)

너가 말한 것처럼 **applicant_count로 통일**이 핵심.

- applicantCount는 **지원자 수**로 취급 (작성자 포함 X)
    
- 계산 기준(권장):
    
    - `recruit_application`의 row count 기반
        
    - 또는 status 조건을 걸지 여부는 정책 선택(보통 APPLIED/ACCEPTED만 카운트)
        
- `current_count`는 “참여 인원(작성자 포함)” 성격이라 applicantCount와 혼동되기 쉬움
    
    - 그래서 API 응답에 둘 다 내려줄지/하나만 유지할지 팀 합의 필요
        

### 작성자가 지원 가능 여부

지금 DB상으론 가능(작성자 user_id도 unique(recruit_post_id,user_id)에 의해 한 번은 들어갈 수 있음)

- 일반적으로는 “작성자는 지원 못 함”이 자연스러움
    
- 정책 결정 필요:
    
    - (권장) 서비스에서 `if (me == authorUserId) throw 400`로 막기

# 추가 사항항
### 1) “principal=Long” 규약의 전파 범위

- 지금은 `AuthUtil.currentMemberId()`가 **Long/String만 허용**.
    
- 추후 Spring Security 설정에서 `principal`이 `UserDetails`/커스텀 객체로 바뀌면 **전 모듈(Recruit 포함) 인증이 전부 깨짐**.
    
- 팀 규약 문서에 “SecurityContext principal은 user_id(Long)”로 명시 필요.
    

### 2) Controller에서 500이 나오는 “전형적인” 원인 공유

- `PATCH /api/recruits/{id}/status`에서 **Body 없이 요청하면**:
    
    - `@RequestBody` 필수 → null → NPE/IllegalArgument → 500 (ExceptionHandler가 잡아서 “Internal server error”만 내려줄 수 있음)
        
- Postman에서 반드시
    
    - Header: `Content-Type: application/json`
        
    - Body(raw): `{ "status": "CLOSED" }`
        
- 그리고 `RecruitExceptionHandler`가 어떤 Exception을 400으로 매핑하는지 팀에게 공유해야 함(현재는 500으로 뭉개질 수 있음).
    

### 3) 작성자 지원 문제(지금 DB에 실제로 row가 들어간 상태)

- 너가 확인한 것처럼 작성자(user_id=1)가 recruit_post_id=2에 `recruit_application` row를 만들 수 있음.
    
- 이 상태에서 “지원자 목록”이 비거나 이상하게 보이는 이유가:
    
    - (a) 목록 API가 “작성자 전용”인데, 작성자 본인이 지원하면 데이터가 섞이는 문제
        
    - (b) applicantCount 정의와 UI 표시가 꼬일 가능성
        
- **정책 결정을 문서화**: “작성자는 apply 금지” vs “허용”
    
    - MVP에선 보통 “금지”가 안정적.
        

### 4) applicantCount vs currentCount 혼동 방지(프론트/백엔드 계약)

- 프론트 카드에 보이는 숫자 1개를 뭘로 할지 반드시 합의:
    
    - `applicantCount`: 지원 row 수(작성자 제외)
        
    - `currentCount`: 참여 인원(작성자 포함 + ACCEPTED 반영)
        
- 지금 DB 스키마/엔티티에서 `current_count`가 존재하고, 서비스에서 ACCEPTED 시 증가/취소 시 감소를 하도록 설계했었음.
    
- 그런데 “선택지 A(자동 closed)”로 가면서 **카운트 정의까지 같이 정리해야 함**.
    

### 5) 스케줄러(자동 CLOSED) 구현 시 주의점

- 스케줄러는 “deadline 지난 OPEN 글을 CLOSED로 변경”인데,
    
    - 동시성: 사용자 요청(update/apply)과 충돌 가능
        
    - 해결: **DB update 쿼리 1방**으로 처리 추천
        
        - 예: `update recruit_post set status='CLOSED' where status='OPEN' and deadline < current_date`
            
- 그리고 “deadline = 오늘”을 CLOSED로 볼지( `<` vs `<=`)도 정책 확정 필요.
    

### 6) Flyway 운영 규칙(사고 방지용)

- **이미 적용된 Vn 파일 수정/삭제 금지**
    
- 누군가 V3를 지우거나 파일명을 바꾸면:
    
    - 로컬/서버 DB의 `flyway_schema_history`와 파일 시스템이 불일치 → 바로 장애
        
- 팀 룰로:
    
    - “마이그레이션은 append-only”
        
    - “수정이 필요하면 V(n+1)로 보정 스크립트 추가”
        
- 그리고 개발 DB 초기화용 SQL은 별도 문서로 제공(운영 DB에서 절대 실행 금지).
    

### 7) Notion에 꼭 남겨야 하는 “API 호출 규약”

- status/decision 같은 Enum 입력은 항상 대문자 문자열로 통일:
    
    - `OPEN`, `CLOSED`
        
    - `APPLIED`, `ACCEPTED`, `REJECTED`
        
- skills는 List로 받지만 **검색 파라미터는 CSV 1개로 받는 규약**(예: `skills=Python,FastAPI`)
    
- apply/cancel은 멱등이므로 프론트에서 중복 클릭/재시도 전략이 단순해진다는 점 공유.
    

### 8) “nickname 추가” 후 Recruit에 미치는 영향(연동 포인트)

- RecruitDetailResponse의 `authorNickname` 채우려면:
    
    - member 모듈에서 `user_id -> nickname` 조회 필요
        
- ApplicationResponse의 `nickname`도:
    
    - 지원자 user_id 리스트를 IN 조회로 배치로 가져와 매핑(N+1 방지)
        
- 그래서 “member 조회를 recruit가 직접 repo로 할지”
    
    - 같은 모놀리식이면 repo 사용 가능
        
    - 분리 모듈/레이어 규칙이 있으면 `MemberService`를 통해 호출
        

### 9) 예외/응답 포맷 통일

- 지금은 `{"message":"Internal server error"}` 같은 단순 포맷이 보임.
    
- 프론트/QA를 위해 최소한 아래는 통일 권장:
    
    - 400: 정책 위반(마감/권한/값 검증) → message + code(optional)
        
    - 401/403: 인증/인가
        
    - 404: recruit/application not found
        
- 특히 500을 줄이고 400으로 내리는 매핑이 중요(디버깅 비용 절감)

### 7) 모집 상태 변경 (A) — **너가 겪은 500 재현/해결 확인**

#### 요청(정상)

- `PATCH {BASE}/api/recruits/1/status`
    
- Header: `Content-Type: application/json`
    
- Body
    

```json
{ "status": "CLOSED" }
```

#### 예상 정상응답

- status가 CLOSED로 바뀐 RecruitDetail/RecruitSummary 형태 응답(구현에 맞는 DTO)
    

#### 500 재현 케이스(너 스샷 상황)

- Body 없이 요청하면 500 가능  
    → 이건 “Postman 요청 실수 + 서버가 400으로 잘 못 떨어뜨림” 둘 중 하나.

### 8) CLOSED 이후 행위 차단 확인

1. `POST /api/recruits/1/apply` → 400
    
2. `DELETE /api/recruits/1/apply` → 400 (정책이 “closed 취소도 금지”면)

## “오류 출력값”을 어떻게 봐야 하는지(포인트만)

- Postman 500이면 **서버 콘솔 로그**에서 실제 스택트레이스를 봐야 함.
    
- 특히 아래는 거의 500 원인 Top:
    
    - `@RequestBody` null
        
    - DTO 필드명 오타(getSkill vs getSkills)
        
    - enum 파싱 실패를 400으로 못 잡음
        
    - Auth principal 타입 불일치(Long 아닌 객체)
        

---

## 사실확인 2회

1. **status 변경 요청을 Body 없이 보내면 500이 날 수 있음**: 네 Postman 스샷에서 PATCH에 body가 없고 500이 실제 발생했음.
    
2. **지원자 목록 API 응답은 PageResponse 형태(items/page/size/totalElements/totalPages/hasNext)**: 네가 올린 지원자 목록 응답 스샷이 그 구조였음.
