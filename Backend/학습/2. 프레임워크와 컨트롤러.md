##### 라이브러리 : 내 코드가 주인 -> 흐름 제어권이 내 코드에 있음
* LocalDataTime.now() / ArrayList.add()
##### 프레임워크 : 자체로 주인 -> 흐름 제어권이 프레임워크에 있음(제어의 역전)
* 서버 프로그램을 만들기 위해 우리가 코드 안에 포함시켜 사용하는 기반 코드
#### Spring Boot(프레임워크)의 역할
- 내장 웹 서버 제공(HTTP 서버) : Tomcat 자동으로 띄움 -> 서버 설치 안해도 됌
- 요청 분기(라우팅) : /health, /rooms 각각 접근 시 실행할 메서드 지정
	-> 이 매핑을 자동 처리
* 객체 관리 : 만든 클래스를 자동으로 생성/관리 -> 필요하면 알아서 주입
* 설정 자동화 : application.propetries기반, 관례에 의한 설정

#### 컨트롤러의 역할 : HTTP <-> 자바 객체/메서드 상호 번역기
컨트롤러가 없으면 서버는 켜져있지만 아무 URL도 처리하지 못함(404 whitelabel)
1. HTTP 요청(TRL, 파라미터, 바디) 수신
2. 자바 메서드 호출 형태로 변환
3. 결과를 HTTP 응답(JSON 등)으로 변환 (어노테이션으로 표기)
##### @RestController : @controller + @ResponseBody
```
@RestController
public class HealthController { ... }
```
	1. 이 클래스는 HTTP 요청을 처리할 수 있다
	2. 메서드의 반환값을 HTTP 응답 바디로 바로 써도 된다

##### @GetMapping : HTTP GET 요청이 ~경로로 들어오면 이 메서드 실행
```
@GetMapping("/health")
public String health() { ... }
```
	* 요청 분기 조건 설정 : HTTP 메서드 : Get, URL 경로 : /health
	-> Spring이 "GET /health -> HealthController.health()" 매핑테이블 생성

##### @PostMapping : HTTP POST 요청이 /messages로 오면 메서드 실행
```
@PostMapping("/messages")
public void sendMessage(...) { ... }
```

##### 컨트롤러가 다루는 것 -> 직접 로직 X, 교통정리
1. URL
2. HTTP 메서드(GET, POST)
3. 요청 파라미터
4. JSON 요청 / 응답
5. 상태코드(200, 404, 500

##### 컨트롤러가 직접 일을 하게되면? 
1. 객체 생성의 주도권이 컨트롤러에게 감 : 컨트롤러의 역할 비대, 역할 분리 불가
2. 테스트가 거의 불가능 : mock 객체 사용 불가, DB 안쓰는 테스트 불가, Service 내부 구현 바뀌면 컨트롤러도 같이 수정
3. 객체 생명주기 통제 불가 : 요청 올 때마다 new UserService() 객체 생성
	-> 메모리 낭비, 상태 관리 불가, 트랜잭션 관리 불가
4. Spring 기능 거의 사용 불가 : Spring이 만든 객체일 때만 작동
=> new를 쓰는 순간 이 객체는 Spring이 아니라 내가 관리한다

#### /health API 흐름 정리
```
http://localhost:8080/health
```
	이 요청이 들어오면
	1. 브라우저가 HTTP GET 요청 전송
	2. Tomcat(내장 서버)이 요청 수신
	3. Spring이 URL(/health) 분석
	4. @GetMapping("/health") 붙은 메서드 탐색
	5. 해당 메서드 실행
	6. 반환 값을 HTTP 응답으로 변환
	7. 브라우저에 출력
	-> 위 과정을 모두 프레임워크가 자동적으로 수행
