> 목표: **“프론트가 뭘 보내고, 백이 어떻게 받고 처리해서 다시 돌려주는지”**를 한 흐름으로 이해하고,  
> 그 과정에서 등장하는 용어(HTTP/JSON/DTO/Jackson/JVM/JIT/컴파일러/인터프리터/Enum/에러규약)를 **초보 기준으로 완전 정리**한다.

## 0. 전체 큰 그림(한 장 요약)

### ✅ 웹앱에서 프론트–백이 연결되는 방식

- **프론트(React)**: 화면(UI) 만들고, 사용자의 입력을 받아 **HTTP 요청**을 보냄
    
- **백(Spring Boot)**: HTTP 요청을 받고, 로직 처리 + DB 작업을 한 뒤 **HTTP 응답(JSON)**을 돌려줌
    
- 둘이 공유하는 “공식 언어”는:
    
    - **HTTP**: 통신 규칙(요청/응답 형식)
        
    - **JSON**: 데이터 표현 형식(텍스트 기반 데이터 구조)
        
    - **API 명세서(계약서)**: URL/메서드/요청·응답 구조/에러 포맷을 합의한 문서

## 1. HTTP & JSON — “어떻게 말하고, 뭘 담는가”

### 1-1. HTTP란?

**HTTP(HyperText Transfer Protocol)**는  
👉 **브라우저(클라이언트)와 서버가 통신하는 규칙(프로토콜)**이다.

#### HTTP 요청은 보통 이런 요소로 구성됨

- **URL**: 어디 기능을 호출할지 (`/api/mates`)
    
- **Method(메서드)**: 무엇을 할지
    
    - `GET`: 조회
        
    - `POST`: 생성
        
    - `PUT/PATCH`: 수정
        
    - `DELETE`: 삭제
        
- **Headers(헤더)**: 부가정보 (인증, 데이터 타입 등)
    
- **Body(바디)**: 보내는 데이터(주로 JSON)
    

예시:

```http
POST /api/recruits HTTP/1.1
Host: api.example.com
Content-Type: application/json
Authorization: Bearer <token>

{"title":"스터디 모집","headcount":3}
```

---

### 1-2. JSON이란?

**JSON(JavaScript Object Notation)**은  
👉 **데이터를 문자열로 표현하는 포맷(data format)**이다.  
프로그래밍 언어가 아니라 “데이터 표기 규칙”.

예시:

```json
{
  "name": "홍길동",
  "age": 23,
  "part": "BACKEND"
}
```

#### “홍길동은 자연어잖아?”

맞다. 하지만 컴퓨터 입장에서는:

- `"홍길동"`도 결국 **UTF-8 같은 문자 인코딩을 거친 바이트(숫자들)**로 저장/전송된다.
    
- JSON은 실행되는 코드가 아니라 **데이터**라서 “컴파일/인터프리트” 대상이 아니다.
    
- JSON은 **파싱(parsing)**해서 “값”으로 꺼내 쓰는 대상이다.
    

---

## 2. 프론트–백 연동의 실제 흐름(요청 → 처리 → 응답)

### 2-1. 프론트가 하는 일

- 사용자가 입력한 값으로 JSON을 만들어서 서버로 보냄
    
- (토큰 방식이면) 요청 헤더에 토큰을 붙여서 보냄
    
- 서버 응답을 받아 화면을 렌더링/분기함
    

---

### 2-2. 백이 하는 일(Controller → Service → Repository)

백엔드는 보통 이런 계층으로 나눔:

- **Controller**: HTTP 요청 입구(라우팅, 간단 검증, DTO 받기)
    
- **Service**: 비즈니스 로직(권한, 규칙, 트랜잭션)
    
- **Repository**: DB 접근(JPA/SQL)
    

---

## 3. DTO & Entity — “왜 굳이 나누는가”

### 3-1. DTO란?

**DTO(Data Transfer Object)**는  
👉 **계층(레이어) 사이에서 데이터를 옮기는 ‘그릇(구조)’**이다.

- Request DTO: 프론트가 보낸 요청 데이터를 담는 그릇
    
- Response DTO: 프론트에 내려줄 응답 데이터를 담는 그릇
    

> DTO가 “변환을 수행하는 도구”라기보다는,  
> **변환 결과물이 들어갈 ‘형식(스키마)’**을 제공하는 타입이다.

---

### 3-2. Entity란?

**Entity(JPA Entity)**는  
👉 **DB 테이블과 매핑되는 서버 내부 모델**이다.  
DB 구조를 많이 따라가며, 내부 로직/관계가 포함될 수 있다.

---

### 3-3. DTO ↔ Entity 흐름(표준)

1. 프론트 JSON → **Request DTO**로 변환
    
2. Service에서 DTO 기반으로 **Entity 생성/수정**
    
3. 결과 Entity → **Response DTO**로 변환
    
4. Response DTO → JSON으로 응답
    

---

### 3-4. Entity를 그대로 반환하면 생기는 사고(현실 사례)

#### (A) 보안/내부 정보 노출

Entity에 `passwordHash`, `refreshToken`, `role` 같은 게 있으면  
실수로 그대로 반환할 때 바로 사고.

#### (B) DB 구조 변경이 프론트 깨뜨림

Entity는 DB 구조 변경에 따라 자주 바뀜.  
Entity를 API로 노출하면 프론트가 DB 변경에 직격탄.

#### (C) 양방향 연관관계 무한 직렬화(무한루프)

- `UserEntity`가 `projects`를 갖고
    
- `ProjectEntity`가 `owner(User)`를 가지면
    
- JSON으로 바꿀 때: User → Project → User → Project… 반복
    

✅ DTO는 “일방향 구조”로 필요한 것만 담아서  
**순환 참조를 설계 단계에서 끊어버린다.**

---

## 4. Jackson & 파싱 & 직렬화/역직렬화 — “JSON이 객체로 바뀌는 과정”

### 4-1. 파싱(parsing)

문자열을 규칙에 따라 분석해서 구조화된 데이터로 바꾸는 작업.  
JSON 파싱은 “코드 실행”이 아니라 “데이터 해석”이다.

---

### 4-2. Jackson이란?

**Jackson**은 Java에서 가장 대표적인 JSON 라이브러리.  
Spring Boot에서 `@RequestBody`를 쓰면 보통 Jackson이 자동으로:

- JSON → DTO 객체(역직렬화)
    
- DTO 객체 → JSON(직렬화)
    

---

### 4-3. “Spring이 title 있는지 검색해서 메서드를 고르나?”

❌ 아니다.

- 어떤 컨트롤러 메서드가 호출될지는 주로:
    
    - URL
        
    - HTTP Method(GET/POST…)
        
    - 매핑 애노테이션(@GetMapping 등)  
        으로 결정됨.
        

그 다음에 Jackson이:

- JSON key `"title"` ↔ DTO 필드 `title`  
    을 **매칭**해서 값을 채운다(데이터 바인딩).
    

---

## 5. 객체(Object), 인스턴스(Instance), 필드(Field), 프로퍼티(Property), 어트리뷰트(Attribute)

### 5-1. 객체(Object)

“데이터 + 구조(필드) + 동작(메서드)”을 묶은 개념.  
실행 중 메모리에 존재하는 “실체”.

### 5-2. 인스턴스(Instance)

클래스(설계도)로부터 만들어진 실제 객체(메모리 상 실체).

- `ProfileUpdateRequest`(타입/클래스)
    
- `new ProfileUpdateRequest("홍길동", 23)`(인스턴스)
    

### 5-3. 필드(Field)

클래스 내부의 멤버 변수(저장 공간).  
Java에서 `private String name;` 같은 것.

### 5-4. 프로퍼티(Property)

“외부에서 보이는 속성”이라는 의미로 쓰이며,  
JavaBeans 관점에서는 getter/setter로 접근되는 논리적 속성.  
실무에서는 field와 섞어 말하는 경우도 많음.

### 5-5. 어트리뷰트(Attribute)

문맥에 따라 다름:

- 일반: 속성(property)
    
- DB: 컬럼(column)을 attribute라고 부르기도 함
    
- HTML: 태그 속성(attribute)
    

📌 결론:

- DTO 전체는 “객체/인스턴스”
    
- 그 안의 `name`, `age`는 보통 “필드/프로퍼티”
    
- DB는 보통 “컬럼(column)”이 정확한 용어
    

---

## 6. Enum — 값 고정의 의미 & 한글 가능 여부

### 6-1. Enum이란?

미리 정한 값들만 들어오게 강제하는 타입.

예:

- Part: BACKEND, FRONTEND, PM…
    

### 6-2. Enum을 왜 쓰나?

- 오타/표기 차이 방지 (`"BackEnd"`, `"backend"` 같은 혼란 제거)
    
- 필터/검색/통계 처리 안정화
    
- 프론트/백/DB의 계약 안정화
    

### 6-3. Enum을 한글로도 가능?

기술적으로 가능하지만 **협업/운영 관점에서 비추천**.

- 표준은 영문 대문자 enum
    
- 화면에 보여줄 때만 한글로 “표시 변환”하는 게 일반적
    

---

## 7. 에러 응답 규약 — 왜 “포맷”을 맞춰야 하나?

### 7-1. 왜 중요한가?

프론트는 에러 상황에 따라:

- 로그인 페이지로 보내기
    
- 권한 없음 안내
    
- 입력칸 오류 표시
    
- 재시도/토스트 알림 등  
    “행동”이 달라져야 한다.
    

그래서 단순한 메시지 문자열만으로 분기하면 코드가 난장판 됨.

---

### 7-2. 추천 에러 포맷(계약)

```json
{
  "errorCode": "AUTH_401",
  "message": "로그인이 필요합니다",
  "fields": {
    "title": "필수 입력입니다"
  }
}
```

- `errorCode`: 프론트 로직 분기용(반드시 고정)
    
- `message`: 사용자 안내(일관성 필요)
    
- `fields`: 검증 오류 표시용(옵션)
    

---

## 8. OAuth & “왜 서버에서 도메인 검증해야 하나”

### 8-1. 프론트만 믿으면 왜 위험?

프론트 코드는 사용자 PC에서 실행되므로 조작 가능:

- 개발자도구로 JS 실행/변수 변경
    
- API 요청을 복사해서 Postman/curl로 서버에 직접 호출
    
- UI에서 막아도 서버로 직접 때리면 우회
    

즉 프론트 검증은 “UX”일 뿐, 보안의 최종 판단이 될 수 없음.

---

### 8-2. 서버에서 어떻게 막나?

서버는 “프론트가 준 이메일”을 믿지 않고,  
**구글이 인증한 결과를 서버가 직접 확인**한다.

권장 흐름(Authorization Code Flow):

1. 사용자가 구글 로그인(구글 페이지에서)
    
2. 구글이 서버 콜백으로 code 전달
    
3. 서버가 code로 구글에 요청해 이메일 등 사용자 정보를 직접 받음
    
4. 서버가 `@catholic.ac.kr` 도메인 검사
    
5. 통과 시 JWT 발급(또는 쿠키 설정)
    

---

## 9. 토큰(JWT/Access/Refresh) & 저장 위치(localStorage vs HttpOnly cookie)

### 9-1. JWT란?

토큰의 “형식(포맷)” 중 하나.  
Access Token을 JWT로 만들 수도 있음.

### 9-2. Access Token

- API 요청 때마다 보내는 출입증
    
- 유효기간 짧게(보통)
    
- 탈취 시 유효기간 동안 악용 가능
    

### 9-3. Refresh Token

- Access 만료 시 새 Access 발급받는 열쇠
    
- 유효기간 길어서 더 위험 → 보관/폐기 정책 필요
    
- 구현 난이도 상승(DB 저장/로테이션 등)
    

---

### 9-4. 토큰 저장 위치가 왜 중요?

토큰이 탈취되면 “그 사람이 너로 로그인 가능”이기 때문.

#### localStorage

- 장점: 구현 쉬움, axios 헤더 붙이기 편함
    
- 단점: XSS로 토큰 읽혀서 탈취 가능
    

#### HttpOnly Cookie

- 장점: JS가 토큰에 접근 불가 → XSS로 “토큰 값 탈취” 어려움
    
- 단점: 쿠키 자동 전송이라 CSRF 이슈 → SameSite/CSRF 방어 필요, 세팅 난이도↑
    

---

### 9-5. HttpOnly 쿠키의 동작(오해 교정)

- 토큰을 URL에 붙이는 방식 ❌
    
- 서버가 응답에서 `Set-Cookie`로 쿠키를 저장하게 함
    
- 이후 브라우저가 요청할 때 자동으로 `Cookie: ...` 헤더에 실어 보냄
    
- HttpOnly면 JS에서 `document.cookie`로도 안 보임

## 10. 컴파일러 vs 인터프리터 vs JVM 바이트코드 vs JIT

### 10-1. 컴파일러(Compiler)

코드를 먼저 번역해 실행 가능한 형태를 만든다.

- Java: `.java → .class(바이트코드)` (javac)
    
- C/C++: 실행파일(기계어)
    

### 10-2. 인터프리터(Interpreter)

코드를 실행하면서 해석(또는 단계적으로) 실행한다.

### 10-3. Java는 혼합 모델

1. **컴파일 단계 존재**: `.java → .class(바이트코드)`
    
2. 실행은 JVM이 함:
    
    - 바이트코드를 해석 실행(인터프리터)
        
    - 자주 쓰는 부분을 기계어로 바꿔 캐싱(JIT)
        

📌 중요한 교정:

- “실행할 때마다 javac가 돈다” ❌  
    (대부분 배포 시 이미 컴파일된 jar/class를 실행)
    
- “바이트코드 이후 JVM이 어떻게 실행할지”는  
    인터프리터 + JIT 혼합으로 진행되는 경우가 많음

- `@RequestParam`: `?part=BACKEND` 같은 쿼리스트링
    
- `@RequestBody`: JSON 본문(body)

## 12. 협업에서 “맞춰야 하는 계약(필수 합의 목록)”

프론트–백 연동에서 반드시 고정해야 하는 것들:

1. **API URL & Method**
    

- `/api/mates` + GET 같은 형태
    

2. **Request JSON 구조**
    

- 필드명, 타입, 필수 여부
    

3. **Response JSON 구조**
    

- 프론트가 그대로 사용하므로 필드명 변경은 파괴적 변경
    

4. **Enum 값**

5. **에러 포맷**
    

- errorCode/message/fields 등
    

이걸 문서로 고정한 게 **API 명세서(스웨거)**이고,  
백엔드가 초안을 잡고 프론트와 합의해서 고정하는 게 가장 효율적이다.

## 14. 용어 미니 사전(자주 튀어나오는 것들)

- **MVP**: 최소 기능 제품(완성 가능한 최소 범위)
    
- **CRUD**: 생성/조회/수정/삭제
    
- **직렬화(Serialization)**: 객체 → JSON
    
- **역직렬화(Deserialization)**: JSON → 객체
    
- **파싱(Parsing)**: 문자열을 규칙에 따라 구조화
    
- **온보딩(Onboarding)**: 새 팀원/사용자가 시스템에 익숙해지게 하는 과정
    
- **CORS**: 다른 도메인 간 요청 허용 규칙(브라우저 보안 정책)
    
- **XSS**: 악성 스크립트 실행 취약점(localStorage 탈취 위험)
    
- **CSRF**: 사용자의 로그인 상태를 악용해 원치 않는 요청 유도(쿠키 자동 전송 위험)
    
- **JIT**: 실행 중 자주 쓰는 코드를 기계어로 바꿔 캐싱해 성능 향상
