#### 공통 인프라 선구현 항목
1. 응답 포맷 통일(JSON)
    - 목표
        - 모든 성공/실패 응답의 형태를 일관되게 유지
        - 클라이언트에서 공통 응답 처리 로직 작성 가능
            
    - 보편 설계
        - 성공: `{ success: true, data: ... }`
        - 실패: `{ success: false, error: { code, message } }`
            
2. 예외 처리 표준화(에러 코드/메시지)
    - 목표
        - 컨트롤러마다 try-catch 반복 제거
        - 에러 응답 포맷과 HTTP Status를 전역적으로 통일
            
    - 보편 설계
        - ErrorCode(enum): `code`, `httpStatus`, `message` 정의
        - 커스텀 예외 클래스에서 ErrorCode 참조
        - GlobalExceptionHandler(@RestControllerAdvice)로 일괄 처리
            
3. 로깅(SLF4J)
    - 목표
        - 요청 처리 흐름 및 에러 원인 추적 가능
        - 운영 환경에서 장애 분석 근거 확보
            
    - 적용 포인트
        - Controller / Service 계층에서 로그 사용
        - 에러 발생 시 stack trace 기록
        - 운영 환경에서는 민감 정보(토큰, 비밀번호, 개인정보) 로그 출력 금지
            
4. DB 트랜잭션 관리(서비스 계층)
    - 목표
        - “게시물 생성 + 첨부 정보 저장”과 같은 복합 작업의 원자성 보장
        - 데이터 불일치 및 중간 실패 방지
            
    - 보편 설계
        - Service 계층에 `@Transactional` 적용
        - 조회 전용 로직은 `@Transactional(readOnly = true)`로 분리
            
#### JPA Auditing 및 시간대 설정
JPA Auditing
- 목표
    - createdAt / updatedAt 컬럼을 모든 엔티티에 자동 적용
        
- 구성
    - `@EnableJpaAuditing` 활성화(Application 또는 Config 클래스)
    - BaseTimeEntity(createdAt, updatedAt) 정의 후 엔티티에서 상속
        
- 기대 효과
    - 데이터 생성/수정 시점 자동 관리
    - 운영, 디버깅, 감사(Log/이력) 관점에서 유리

시간대 고정(Asia/Seoul)
- 목표
    - 서버, DB, 로그, 스케줄러의 시간 기준을 통일
        
- 보편 구현 위치
    - Application 시작 시점(@PostConstruct 등)에서  
        `TimeZone.setDefault(TimeZone.getTimeZone("Asia/Seoul"))` 설정
        
- 주의
    - 운영 서버 OS 시간이 UTC여도 애플리케이션 기준 시간은 고정 가능
        

#### 스케줄러 기반 운영 작업
- 목표
    - 운영성 기능을 개발 초기에 구조만이라도 확보
    - 이후 RefreshToken / 임시 데이터 정책에 따라 실제 로직 확장
        
- 보편적인 스케줄 작업 예시
    1. 만료 RefreshToken 정리(RefreshToken 도입 시)
    2. 임시 업로드 파일 정리(Upload 기능 도입 시)
    3. 통계 집계(게시물/좋아요/댓글 수 일별 집계 등)
        
- 구성
    - `@EnableScheduling` 활성화
    - `@Scheduled(cron = ...)` 또는 `fixedDelay / fixedRate` 방식 사용
        

#### 로컬 실행 검증
1. Gradle 동기화 성공
    - `build.gradle` 의존성 다운로드 및 빌드 성공 확인
        
2. 애플리케이션 실행 성공
    - IntelliJ Run 또는 `./gradlew bootRun`
    - 콘솔에 서버 시작 로그(Tomcat started, port 등) 출력 확인
        
3. DB 연결 확인
    - local 프로필에서 H2 / MySQL 접속 성공
    - (H2 사용 시) H2 Console 사용은 학습 편의상 선택
        
4. 기본 엔드포인트 확인(임시)
    - Actuator 사용 시 `/actuator/health`
    - 미사용 시 헬스 체크용 테스트 컨트롤러 임시 구성

#### 배포 기본 뼈대
- 목표
    - 운영 배포를 후순위로 미루지 않고  
        jar 실행 + start/stop 스크립트 + appspec.yml 구조를 초기에 확정
        
- 구성 요소
    1. 단일 Jar 산출
        - `./gradlew build` → `build/libs/*.jar`
    
    2. scripts/start.sh    
        - 역할: 새 Jar 실행(백그라운드 실행, 로그 파일 지정 등)
    
    3. scripts/stop.sh    
        - 역할: 기존 프로세스 종료(포트 또는 프로세스 기준)
    
    4. appspec.yml    
        - 역할: 배포 단계별 스크립트 실행 순서 정의
        - 예: `BeforeInstall`, `AfterInstall`, `ApplicationStart` 훅
